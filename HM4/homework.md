# Домашнее задание №4: Проектирование Notification Platform

## Контекст
Централизованная платформа уведомлений для онлайн-банка, обеспечивающая маршрутизацию, гарантированную доставку и управление приоритетами для транзакционных, сервисных и маркетинговых сообщений через каналы Push, SMS и Email.

---

## Блок 1. Базовая архитектура

### 1. Функциональные требования
1. **Единая точка входа (API):** Система должна предоставлять внутренним микросервисам банка единый API для отправки уведомлений всех типов, инкапсулируя логику работы с конкретными провайдерами (APNs, Firebase, SMS-шлюзы).
2. **Управление каналами и Fallback:** Система должна уметь автоматически переключаться на резервный канал доставки (например, с Push на SMS), если основной канал недоступен или уведомление не доставлено в течение заданного таймаута.
3. **Управление предпочтениями (Opt-out):** Пользователи должны иметь возможность отключать маркетинговые и сервисные уведомления. Транзакционные уведомления являются критичными и отключению не подлежат.
4. **Приоритизация сообщений:** Система обязана распознавать тип уведомления и обрабатывать транзакционные сообщения вне очереди массовых маркетинговых рассылок.
5. **Трекинг статусов:** Система должна сохранять историю изменения статуса каждого уведомления (создано, отправлено провайдеру, доставлено, ошибка) для разбора инцидентов службой поддержки.

### 2. Нефункциональные требования
1. **Производительность (Latency):** 99-й перцентиль (p99) времени обработки транзакционного уведомления (от получения API-запроса до передачи во внешний шлюз) не должен превышать **2 секунд**.
2. **Масштабируемость (Scalability):** Система должна выдерживать пиковые нагрузки до **1 000 000 уведомлений в час** при запуске маркетинговых кампаний без влияния на Latency транзакционных сообщений.
3. **Доступность (Availability):** Уровень доступности API приема уведомлений должен составлять **99.99%**, так как его недоступность блокирует критичные процессы банка (переводы, оплаты).
4. **Надежность (Reliability):** Система должна обеспечивать гарантию доставки **At-least-once** для транзакционных уведомлений, исключая потерю сообщений при рестартах или падениях узлов системы.
5. **Безопасность (Security):** Персональные и финансовые данные в уведомлениях должны маскироваться при логировании. Передача данных внешним провайдерам должна осуществляться только по зашифрованным каналам (TLS 1.2+).

### 3. Архитектурно значимые требования (ASR)
| ASR | Связанные требования (FR/NFR) | Влияние на архитектуру |
| :--- | :--- | :--- |
| **ASR-1: Строгая изоляция нагрузки (Traffic Segregation)** | Latency ≤ 2 сек, Масштабируемость (1 млн/час), Приоритизация. | Требует физического или логического разделения потоков данных (отдельные топики/очереди в Message Broker) и независимых пулов обработчиков, чтобы маркетинговый спайк не замедлил транзакционные отправки. |
| **ASR-2: Отказоустойчивость и Fallback** | Управление каналами, Доступность 99.99%, Гарантия At-least-once. | Исключает синхронное взаимодействие с провайдерами. Требует внедрения асинхронных очередей, планировщика задержек (Delay Queue / Timer) и хранилища состояний доставки для реализации паттерна Circuit Breaker и механизма Retry. |
| **ASR-3: Идемпотентность обработки** | Гарантия доставки, Снижение жалоб на дубликаты. | Требует внедрения быстрого Key-Value хранилища (In-memory) для проверки уникальности Idempotency Key (ID транзакции) перед каждой отправкой, чтобы защитить пользователей от двойных списаний при сетевых ретраях. |

## Блок 2. Развитие архитектуры (4–7 баллов)

### 4. Ключевые архитектурные вопросы
1. **Вопрос:** Как обеспечить независимое масштабирование транзакционных и маркетинговых потоков?
   * **Порождающие требования:** ASR-1 (Изоляция нагрузки).
   * **Почему важен:** Если рассылка забьет общую очередь или исчерпает соединения с БД, пользователи не получат коды подтверждения (нарушение SLA), что приведет к финансовым рискам для банка.
2. **Вопрос:** Какова стратегия оркестрации при отказе основного канала (например, APNs для iOS)?
   * **Порождающие требования:** ASR-2 (Отказоустойчивость).
   * **Почему важен:** Необходимо найти баланс между стоимостью и задержкой. Моментальный переход на SMS дорого обойдется бизнесу, а долгое ожидание восстановления Push-канала нарушит SLA 2 секунды.
3. **Вопрос:** Как реализовать дедупликацию уведомлений в распределенной среде?
   * **Порождающие требования:** ASR-3 (Идемпотентность).
   * **Почему важен:** При сбоях сервисы-источники будут повторять запросы. Без распределенного механизма дедупликации клиент может получить несколько SMS об одной операции.

### 5. Архитектурные последствия ASR
* **Для ASR-1 (Изоляция нагрузки):** Внедрение Message Broker (Kafka или RabbitMQ) с жестко выделенными ресурсами под транзакционные топики. Развертывание независимых микросервисов-воркеров для каждого типа трафика.
* **Для ASR-2 (Отказоустойчивость):** Использование паттернов Circuit Breaker для интеграции с внешними API. Внедрение базы данных (PostgreSQL/MongoDB) для хранения стейт-машины уведомления (статусы переходов) и cron/timer-воркеров для запуска Fallback-сценариев.
* **Для ASR-3 (Идемпотентность):** Добавление Redis-кластера. Каждый входящий запрос должен содержать уникальный ключ, который проверяется и атомарно блокируется в Redis перед постановкой задачи в брокер.

### 6. Архитектурные решения, которые НЕ подходят
1. **Синхронные вызовы внешних провайдеров (HTTP REST напрямую из сервиса приема)**
   * **Какой ASR нарушается:** ASR-2 (Отказоустойчивость).
   * **Почему:** Если SMS-шлюз будет отвечать 5 секунд или упадет, поток в нашем API заблокируется. Это приведет к каскадному отказу (exhaustion of connection pool) и падению всей платформы уведомлений банка.
2. **Единая монолитная реляционная БД для хранения тела сообщений и логов статусов**
   * **Какой ASR нарушается:** ASR-1 (Изоляция нагрузки).
   * **Почему:** Запись 1 миллиона маркетинговых сообщений и постоянные апдейты их статусов исчерпают IOPS базы данных. Транзакционные запросы встанут в очередь блокировок на уровне БД.

### 7. Неопределённости и архитектурные риски
1. **Неопределённость:** Строгие лимиты (Rate Limits) и реальная стабильность внешних провайдеров SMS/Email. 
   * **Как проверить:** Проведение нагрузочного тестирования на тестовых контурах провайдеров (если предоставляются) и детальный аудит SLA в юридических договорах с вендорами. Настройка алертинга на метрики 429 Too Many Requests.
2. **Неопределённость:** Реальный процент доставки Push-уведомлений на устройства (влияние фоновых ограничений Android/iOS).
   * **Как проверить:** Внедрение Canary Release (постепенной раскатки) механизма Fallback на небольшой группе пользователей для сбора аналитики: как часто система реально вынуждена переключаться на дорогостоящие SMS. Это позволит скорректировать бюджет на SMS-шлюзы.
